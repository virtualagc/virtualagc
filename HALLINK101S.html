Introduction

A "link editor" (or just "linker") is a program used to combine a set of "object files" created by compilers and assemblers generated from separate source code files into a single executable "load file" that can be directly run on a computer after being loaded into the computer's memory.  In the specific case of Space Shuttle flight software under the umbrella of the Virtual AGC Project, the roughly 1500 source-code files are from the flight software known as PFS or PASS and (potentially) BFS.  They're written in the HAL/S high-level language and the AP-101 assembly language, and compiled/assembled by the HALSFC compiler or ASM101S assembler, thus leaving us with around 1500 object-code files that need to be linked into a single load file.  But even if (say) you had written your own custom HAL/S program, and you did so using a single source-code file, thus creating a single object file, that object file would still need to be "linked" to produce an AP-101S load file.  That's because what distinguishes an object file from a load file is that a load file inhabits a specific area of computer memory at explicit memory addresses, whereas an object file leaves all questions of the specific memory addresses it occupies open; it is the linking process which turns the open-questions of addressing in the object files into the specific memory addresses a program needs if it's going to run properly.

Don Schmidt (thanks, Don!) has been kind enough to write a program he calls lnk101 that we can use for this purpose. 
Disclaimer:  I shouldn't imply that lnk101 was written by or for Virtual AGC.  It was at least begun for Don's own purposes, and thus existed in some form when I first heard mention of it, but Don was kind enough to pay attention to my many ideas on the subject as his development proceeded, so lnk101 is indeed fit for our use within the Virtual AGC suite of Space Shuttle development tools.  But note that it does not reside within Virtual AGC software repositories, nor is Virtual AGC in control of it, nor is anything I say about it official.  It is in Don's own repository, as is public commentary of his own about its use.
Regarding the contemporary linker used for the original Space Shuttle software development, to my knowledge it has not survived, either in source code or otherwise.  Thus lnk101 is an entirely-new creation.  It is a Python 3 program that should work on any computer having the Python 3 language installed.

Regarding the IBM AP-101B vs AP-101S computers, the AP-101B was used in earlier Shuttle flights, while the AP-101S was used in later flights.  I think that lnk101 is agnostic on the question of AP-101B vs AP-101S, since the choice of instruction sets is the province of the HAL/S compiler rather than of the linker.

I am not aware of surviving documentation describing the format of AP-101S object files specifically, nor of any surviving AP-101 object-code files from the Shuttle era.  In any case, I think that such object code would have been produced not as "files" in a mass-storage system, but rather as decks of computer punch-cards punched directly by the original HAL/S compiler (HAL/S-FC) or AP-101 assembler.  So it is possible that the decks of punch cards could still exist, hidden away somewhere, even if not as online files.  In any case, I have no access to them, and have heard nary a hint of their existence, and I could be mistaken in my inferences about them anyway.

The original linker is referred to in the contemporary documentation simply as "the AP-101 link editor", and the contemporary documentation goes on to say that "until the final version of the linkage editor is available to support all features of the HAL/S-FC compiler, an interim program, known as HALLINK-FC, will be supplied to perform the link edit functions".  As usual, satisfactory documentation for neither "the AP-101 link editor" nor for HALLINK-FC (nor for its System/360-target predecessor HALLINK) is available to us.  Indeed, I do not even know if the "the AP-101 link editor" was ever completed satisfactorily to supersede HALLINK-FC. My intuition, based on nothing concrete, is that it was not, and that HALLINK-FC was used until the end of the Shuttle project.  But since we have access to neither of those linkers, any intuition of mine about them is without value.

On the other hand, in the Virtual AGC Project we have created a modern port (from the XPL/I programming language into the Python programming language) of the original HAL/S compiler.  That being the case, we can in principle use the compiler to convert HAL/S source code into AP-101S object files and apply reverse engineering on those object files to clarify issues regarding the object-file format.  In attempting that kind of reverse engineering, it quickly becomes apparent that the AP-101S object-file format is based upon the format of early object files for the IBM System/360, if not strictly identical to it.  In other words, we can use the well-documented System/360 format as our guide, and occasionally depart from it as needed on an ad hoc basis.  Some relevant references are listed in the next section.
Aside:  It appears to me that in IBM parlance, what I call an "object file" would be a referred to as an "object deck", presumably because it was originally directly punched onto a stack of computer punch-cards.  I'm not telling you this just to be pedantic — though obviously, that's a thing I do — but rather to let you know that if you google it yourself, you're likely to have much better luck searching for "decks" instead of "files".

As usual, not all of the System/360 features are needed for actual legacy AP-101S software, so I won't claim that every feature provided by a full System/360 linker is necessarily implemented in lnk101; that's up to Don anyway.  Contrariwise, some non-System/360 AP-101S-specific features appear in the object files.  For example, "records" in a System/360 object file are described as being of 6 types, known as ESD, TXT, RLD, SYM, XSD, and END, but the XSD type wasn't introduced until the 1990's in order to deal with the needs of C compilers, and hence XSD records don't appear at all in HAL/S or AP-101S object files.  But I won't trouble myself or bore you with an exhaustive list of such differences here. 

To facilitate whatever reverse engineering is needed, I've provided a Python 3 program that can parse the object files produced by the HAL/S compiler (or by ASM101S) and report on all of the data found within them; and Don does similarly.  This will be covered in more detail later.

Backup vs Primary Flight Software (BFS vs PFS)

Let's get a clarification to my comments above out of the way.

Recall that Space Shuttle flight software came in two distinct flavors, namely the Primary Flight Software (referred to as PFS, or loosely but more commonly as PASS) and the Backup Flight Software (BFS).  PFS and BFS were developed mostly independently of each other — by different companies even, namely Intermetrics vs Rockwell —, but more than that, their development tools differed as well.  For example, although PFS and BFS nominally were eventually compiled using the same HAL/S compiler (HAL/S-FC), in fact there were two incompatible versions:  HAL/S-FC (for PFS) and HAL/S-BFC (for BFS).   And the fact that there are two separate compilers doesn't affect just PFS and BFS code, but any HAL/S program we may write:  We always have a choice of which of the two flavors of the compiler to use!

More importantly for us in the context of the present discussion, back in the day, PFS and BFS used completely different contemporary linker programs.  Outside of this one section you're reading now, both this entire page and specifically the lnk101 linker program described on this page, are presently for object code generated by the PASS/PFS flavor the HAL/S compiler.  Object code generated by the BFS version of the compiler won't even be mentioned outside of this one section you're reading right now ... well, with one brief return when reverse engineering is discussed later. 

Why?  Well, back in the day, rather than using the AP-101 link editor, BFS was linked using a different program called PILOT (Program Integration and Loading Tool).  Furthermore, while object code produced by the PFS version of HAL/S-FC was in a modified System/360 format, object code produced by HAL/S-BFC was instead in PILOT format.  The two object-code formats are incompatible.  In other words, neither the AP-101 link editor nor its intended replacement lnk101 are compatible with PILOT object files.  At least, not at present.

That said, I hope that someday there will be a modern version of PILOT that can be used to link BFS-style object code ... perhaps it can be called yaPILOT?  Maybe?  Yes, that has a nice ring to it!  But better, I'd prefer that lnk101 be extended in that direction.

One hurdle is that as far as I know, there is no surviving information whatsoever about PILOT, and in particular we have no documentary information about its object-file format.  To be frank, I'm proud of myself for even figuring out that it was called PILOT to begin with.  The acronym "PILOT" is, of course, quite unfortunate from the point of view of trying to find available information via a search engine like Google, since all you'll discover is information about how to pilot a Space Shuttle.  I'm afraid this reveals a certain weakness in today's treatment of knowledge, in that we don't have many knowledgeable people any longer, but rather we have people with borrowed knowledge attained via search engines.  And A.I., of course, for those of us who can't even use search engines very effectively.  But what about when the search engines fail because the search terms are too common?  Well, that's a good way to lose knowledge, even if it's sitting in plain view.  But I doubt it would have helped if PILOT had a less-common name or if we had more native experts on PILOT to consult, since I doubt that there's any surviving information about PILOT online anyway.  If you have any such documentation, send it to me! 

But don't lose all hope.  Since the object-code generator in PASS2 of the HAL/S-BFC source code can produce PILOT-compatible output, it's still possible to attain some understanding of the PILOT format via reverse engineering it, and indeed perhaps a sufficient amount to implement a linker!  I have done at least some of the reverse engineering, and have embedded my findings in a program I call coPILOT.py.  Catchy, eh?  This script can read object files created by the HAL/S-BFC, and parse them into a human readable form.  Moreover, there's additional commentary embedded within coPILOT.py's source code to document some aspects of the format that aren't evident in the reports it outputs.  If you want to read more about coPILOT.py, I'd suggest referring to the discussion in issue #1273 in our Virtual AGC GitHub repository.

And now we leave the topic of BFS in order to resume our discussion of PASS/PFS-style object code and its linker.

Installation of lnk101

As mentioned earlier, the AP-101S linker we have is called lnk101, by Don Schmidt.  To install it, and a variety of useful programs that come with it, I would suggest this:
  • Use the build instructions given by Don Schmidt's software repository.
  • Get the directory nsts-sdl-dps/build/bin into your PATH, or at least copy or symbolically link all of the program in nsts-sdl-dps/build/bin/ into some directory that's already in your PATH.

I know this works on the versions of Linux I personally use.  Concerning Mac OS or Windows I have no knowledge.

Note:  At the present writing, the software in Don's repository is under development, hence the installation process still has what I consider inconveniences that I hope won't persist into a production version.  Inconveniences?  If you are reading this page at all, it's likely you will already have installed all of the Shuttle-related development tools (HALSFC, ASM101S, XCOM-I) on your own computer.  But installing lnk101 using the instructions above will at the moment install duplicates of all of these Virtual AGC development tools, sometimes modified, sometimes behind in features, sometimes ahead in features (which will hopefully be merged back upstream to Virtual AGC sometime soon), all of which will be time-consuming for you, and none of which you will personally use.  But as long as none of as any of these duplications work their way into your PATH, having them there doesn't do you any particular harm.

Use of lnk101

There are a couple of scenarios in which I would envisage using the linker.  I'll cover them both in the subsections below. 

Most commonly, I'd think that you'd want to take an object file or set of object files comprising a complete program, and link them to create a load file that you could run in an AP-101S emulator.  At present, I'm still working towards full compilation/assembly of all PASS/PFS source-code files, so it's not actually possible to do this yet with PASS.  But it's certainly possible to do so with a HAL/S program you've written yourself.  The other potential use is something that I need to do myself, which is to take a single object file that doesn't represent a complete program itself, but to nevertheless link it as if it (and all of the not-actually-present other object files upon which it relies) reside at some specific locations in memory of my own choosing.  That's capability is vital for testing the compiler and assembler.

By the way, if you were immersed in the IBM mainframe world, as you would be if you were one of the original developers of Space Shuttle flight software, you would expect that since the linker intakes IBM System/360-like object files it would also emit IBM System/360-like load files.  That's certainly what the contemporary linked editor did and as the contemporary AP-101S loader would expect.  But lnk101 does not do that.  Instead, it emits what it refers to as an "FCM" file.  (I'm not sure what that stands for.  Perhaps Flight Computer Memory.)  At any rate, it's simply a binary file that's an image of the entire AP-101S memory.  Not coincidentally, this is also the format expected by the available AP-101S emulators, gpc-batch and gpc-gui.

Scenario 1:  Fully Linking a Complete Program, and Running It

Suppose for the sake of discussion that you have a HAL/S program residing in a single source-code file, perhaps HELLO_WORLD.hal:

C A TEST PROGAM
  HELLO_WORLD: PROGRAM;
    WRITE(6) 'Hello, world!';
    WRITE(6) '';
  CLOSE HELLO_WORLD;

In the simplest manner available right now, you might compile it, link it, and run it as the following table describes:

Command-Line Instruction
 Explanation
 
HALSFC -o HELLO_WORLD.obj HELLO_WORLD.hal
 HALSFC is, of course, our HAL/S compiler, whose more-complete usage instructions are found here.
 
lnk101 -o HELLO_WORLD.fcm --json-symbols HELLO_WORLD.sym.json -L path/to/ZCON -L path/to/RUN HELLO_WORLD.obj
 If our program had consisted of several object files that we had compiled separately, rather than just a single file, then we could have listed all of them on the command line.  The option "--json-symbols FILENAME.json" tells the linker to output a symbol table that will be used below by the emuator(s); without it, the emulator(s) will complain.  The "-L LIBRARY" options give the names of "libraries" that will be searched during the linking process when unresolved symbol names for external procedures or variables are encountered.  A "library" is a directory containing AP-101S object files.  The two libraries you almost-definitely need for any linking are the RUN library and the ZCON library, which provide the PASS runtime library and the pointers (ZCON's, discussed later) for them.  Those libraries are built during the lnk101-installation procedure discussed earlier, as built from AP-101 assembly-language source code here and here, and reside in the build directories nsts-sdl-dps/build/runtime/RUN/ and nsts-sdl-dps/build/runtime/ZCON/ respectively.  Typing that every time you want to link something is not very convenient, so personally I just copy them (or symlink to them) in whatever my working directory is, so that I can just use "-L RUN -L ZCON". 
gpc run --interactive HELLO_WORLD.fcm
 There are several alternatives, one of which is this.  The program gpc is an AP-101S emulator and debugger that operates from a command line. It is available as part of the lnk101 installation.  For more information, see the page dedicated to emulation.

Scenario 2:  Linking a Single Object File That Is Not a Complete Program

In this scenario, you have one (or a few) object file(s) that form part of a program, but you don't have 100% of the files that comprise a complete program.  Or you have them but just want to focus our attention on a few of them.  You could still try to link such object files using the technique described in the preceding section, but the code in those files would undoubtedly reference external symbols.  The linker wouldn't know how to resolve those external references, so there wouldn't be much point to the exercise of running the linker in the first place.

But wait!  Perhaps you could provide the linker with a list of all the CSECTs it will encounter, and the memory addresses at which you want those CSECTs to reside.  If the linker had that information, then it could do its job.  

How would a need to do this arise?  Well, imagine the following situation.  Suppose you have a lot of HAL/S and AP-101S assembly-language source code files, comprising some program.  In my case, that happens to be the source code for version OI340600 of the Space Shuttle's PASS.  If I were in a position to compile/assemble every single source-code file in OI340600 with 100% accuracy, and could figure out such details as the order in which the object files needed to be linked, then in principle I could use the preceding section to link that entirety of OI340600 in a single operation.  But even if the HAL/S compiler and the linker works 100% perfectly, that's still a big task, and very time-consuming as well — not to mention that at this writing I've only been able to compile about 2/3 of the HAL/S source code files, and need to resolve some issues to compile the rest of them.  So as a first step, I'd prefer to see the development tools do their thing piecemeal rather than in one huge lump.  In other words, I very much want to deal with just one HAL/S source-code file at a time, and resolve any issues I encounter in that simplified scenario before moving onward and upward to combining 1000+ oebject files.

But how would I know where all of the CSECTs were "supposed to" go in memory?   I have an additional thing, a disassembly of GPC memory contents for OI340600, made during the Shuttle era by a program called MAFGEN.  Such a disassembly can be parsed to obtain a list of all CSECTs and their locations in memory, using a utility I've written called unlinkMAFGEN2.py.  Of course, in a project of your own devising, using your own source-code files, you could produce such a file of CSECTs/addresses of your own, and we'll come back to the subject of the format of that file shortly.  I face that problem myself, in that while I have a MAFGEN disassembly of OI340600, I do not have one for OI301700, which is another PASS version for which I have source code.

Actually, I've ignored an important detail in what I just said, which is that there are multiple possible memory loads of PASS executables.  What I mean by that is that these different memory loads involve different selections of PASS source code, and involve different levels of functionality.  Any given one of the five GPCs may be loaded with any one of these memory loads at any given time, according to the flight phase.  Perhaps the following illustration that represents how the Space Shuttle's flight phases are represented in a software model will make the idea clearer; the different memory loads correspond to the pink boxes in the graphic:



Thus, there is not just one MAFGEN disassembly for OI340600, but rather 8 of them, with naming like "DASS_XXX":
  • For OPS 000, system software:  DASS_SSW_(PostIPL)
  • For OPS 1, Ascent:  DASS_G16
  • For OPS 2, On orbit:  DASS_G2
  • For OPS 2, Orbit/doors:  DASS_S2
  • For OPS 3, Entry:  DASS_G3
  • For OPS 8, On-orbit checkout:  DASS_G8
  • For OPS 9, Precount/post-landing:  DASS_G9
  • For OPS 9, Mass memory utility:  DASS_P9
Similarly, there is not just one list of CSECTs and their memory addresses as produced by unlinkMAFGEN2.py, but rather 8 of them with names of the form "csects-XXX.json":  csects-SSW.json, csects-G16.json, and so on.

To illustrate the linking process, let's consider just a single HAL/S source-code file from OI340600 that I've chosen basically at random, namely APPLSRC/GV7RTL.hal.  Incidentally, GV7RTL appears only in memory load G16.  The steps to compile and link this file might look like this:
HALSFC APPLSRC/GV7RTL.hal --test --force --parms=LIST,SRN,TEMPLATE,NOLFXI,REGOPT,CARDTYPE=FCRM -o GV7RTL.obj
lnk101 GV7RTL.obj -o GV7RTL.fcm -L RUN -L ZCON --json-symbols GV7RTL.json --external-syms csects-G16.json
In other words, from the HAL/S source GV7RTL.hal, produce an object file GV7RTL.obj, and then link GV7RTL.obj to produce an AP-101S memory image file GV7RTL.fcm.  These operations succeed, but ... so what?  Now we've got GV7RTL linked and ready as a memory image, but what can we do with it?  We certainly couldn't run it in isolation, without the rest of PASS, even if we happened to know what GV7RTL is supposed to do!  (At least I don't, and forgive me for projecting, but I carelessly assume that you don't either.)  If this were your own HAL/S project rather than mine, you might just be pleased that it had succeeded at all, and leave it at that.

In the case of OI340600, though, I've left out a crucial detail of significance, which is that beyond just providing us with CSECT addresses, the MAFGEN disassemblies also contain the complete contents of GPC memory in hexadecimal form ... though admittedly it's a bit of a chore to extract it.  But extracting the memory image is valuable in and of itself, since in principle you could run that dumped GPC memory in an AP-101S emulator as-is, without even compiling anything in the first place! 

Did I bury the lede on that?  Should I have revealed earlier that compiling, assembling, and linking PASS source code is an unnecessary waste of time?  No, because it's not!  While we have MAFGEN disassemblies for OI340600, we don't have them for any other PASS or BFS version, and in particular don't have them for OI301700.  So while having the MAFGEN disassemblies is certainly very valuable in the sense of making sure our entire system of Virtual AGC development tools is working fine, it's not a solution for getting an executable version of BFS or for any version of PASS other than OI340600.  If we ever expect OI301700 to run, we'd need all the compilation, assembly, and linking we can get, and it better be 100% exact.  Hopefully after perfecting it with OI340600, it will be!

But I digress!  An additional feature of unlinkMAFGEN.py is that beyond just extracting lists of CSECTs vs addresses, at the same time it also extracts the complete memory dumps (G16.fcm, G2.fcm, and so on), in the form of files compatible with Don Schmidt's emulator.  And not only that, among the tools we can find in Don Schmidt's repository is one he calls fcmcmp.  Given a list of CSECTs, fcmcmp will compare two memory-image files, but only the contents of the CSECTs that are relevant.  In our example of GV7RTL, the memory dump for load G16 as a whole is huge, while that for GV7RTL is small in comparison, but only the 11 CSECTs that end up being included in GV7RTL are actually compared, while the remainder of G16 is simply ignored.
Note:  There are other subtleties that fcmcmp helps us to deal with as well.  In principle, before any programs are loaded into it — i.e., prior to the Initial Program Load (IPL) —, every (halfword) location in AP-101S memory at (halfword) addresses less than 0x20000 is loaded with the default value 0xC9FB, while each location at a higher address is loaded with the default value 0xC6C6.  When a program is loaded into the AP-101S memory, by design it's kind of like a swiss cheese, in that not every memory location in every CSECT is necessarily loaded with a specific value: some locations might simply be left open, and whatever value had been preloaded at the location still "shines through" the opening.  Then too, the IPL process may involve moving around chunks of memory from one location to another.  The net effect is that sometimes an otherwise-uninitialized location in memory may contain a value of 0xC9FB, or it may contain a value of 0xC6C6, or frequently may even have become initialized by the IPL to 0x0000.  But fcmcmp sees through this little trick, and so during the comparison process it will match 0x0000 against either of 0xC6C6 or 0xC9FB without treating them as errors.
What does such a comparison look like?  Well, for GV7RTL, the invocation is like so:
fcmcmp --csect-table csect-G16.json GV7RTL.json GV7RTL.fcm G16.fcm
And the results of the comparison?  Like so:
Note: images differ in size (147478 vs 330394 halfwords)

  OK:   #QMM1DNP @ 004EE (2 halfwords)
  OK:   #ZGV7RTL @ 00516 (2 halfwords)
  OK:   #QMV6DN  @ 00518 (2 halfwords)
  OK:   #QVO6DN  @ 0051A (2 halfwords)
  OK:   #DGV7RTL @ 0B4EC (28 halfwords)

And our conclusions?  At least for GV7RTL.hal and for flight phase OPS 1 (load G16), the HAL/S compiler, the linker, and the extracted memory dump from the GPC via the MAFGEN disassembly and unlinkMAFGEN2.py, all agree perfectly.  The system works!  It only remains to check the other ~1500 or so source-code files vs all 8 memory loads, and our confidence for OI340600 will be complete.

I promised earlier to describe the format of the CSECT/address files, so that you might form one of your own if you wanted to use the partial-linking technique in this section on a project of your own.  This information is in the form of a so-called JSON file that's laid out as
{
  "CSECT1": { ... },
  "CSECT2": { ... },
  "CSECT3": { ... },
  ...
}

where CSECT1, CSECT1, and so on, are the names of the CSECTs.  And not just the CSECTs in the object files you're not supplying, but in the ones you are supplying the linker as well.  As far as the entry for any given CSECT is concerned, it'll be laid out like this:

{
  "start": N1,
  "end": N2,
  "hal": "NAME",
  "type": "TYPE",
  "invariant": B,
  "contents": {
    "SYMBOL1": M1,
    "SYMBOL2": M2,
    "SYMBOL3": M3,
    ...
  }
}

It's valuable to note that the fields in these objects were originally determined by whatever information I thought it was feasible for unlinkMAFGEN2.py to extract from a MAFGEN disassembly, and not necessarily by the minimum amount of information I thought would be needed for linking purposes.  In other words, there's extra stuff here that you wouldn't need for linking, and therefore wouldn't need to add to your own CSECT/address file.  Fields not needed by the linker will be ignored, and not all of the fields shown here will necessarily be present or used by the linker.  Provide what you can, and then react to whatever complaints the linker throws at you!  But any symbol that's referenced by your object code will definitely have to be available within the CSECT/address file or else lnk101 will fail.

  • The "start" field is definitely essential.  N1 and N2 are decimal numbers giving the halfword memory addresses at which the CSECT begins and ends.  By "halfword address", I mean that when the address increases by 1, that refers to a location 2 bytes later in memory.  And by "ends" I mean the last halfword address in the CSECT, not the address immediately following the CSECT.
  • Regarding the "hal" field, CSECT names are invariably given as 8-character strings which may be accurate reflections of names used in assembly-language source-code files, but are often inadequate for objects such as PROCEDUREs or variables defined in HAL/S source-code files.  When available, the "hal" field lists the full HAL/S name of the object the CSECT represents.
  • TYPE is one of "", "BCE", "DATA", "EXCLUSIVE", "HAL_LIBRARY_CODE", "HAL_LIBRARY_DATA", "HAL_LIBRARY_ZCON", "MSC", "NONHAL", "PATCH", "PDE", "PROCEDURE", "PROGRAM", "STACK", "ZCON".  Some of these are probably obvious, while some of them are mysteries to me.
  • B is a boolean, i.e. having either of the values true or false.  It is unclear to me what it means for a CSECT type to be variant vs invariant.
  • The subfields of the "contents" field represent the publicly-accessible symbols in the CSECT.  But for example, if the CSECT represented a PROCEDURE, then variables, labels, or embedded definitions of PROCEDUREs local to that PROCEDURE wouldn't be provided.   M1, M2, etc., are decimal-number halfword offsets into the CSECT, beginning at 0 for the start of the CSECT; i.e., they tell us where in the CSECT the data associated with the symbol can be found.

Of course, the documentation needs to be consulted even to understand the parsed information in the report in any detail.  Nevertheless, even on a superficial reading, this report tells us that the object file contains 11 "records", of type SYM (3), ESD (2), TXT (4), RLD (1), and END (1), and reveals how various features easily visible in the HAL/S source code are represented in the object file.  You may notice that the report doesn't really provide much information about individual SYM (symbol-table) records.  That's because all SYM records need to be conjoined end-to-end, and it is only the conjoined record that contains parsable information; whereas the individual SYM records are not separately parsable.  Parsing the conjoined record is where the "SYM-Record Summary" at the end of the report comes from.

In the reverse-engineering report we see items obviously related in a general way to the HAL/S source code, though the relationship is not immediately clear in detail, such as the symbols #CSTATIS, #ZSTATIS, #DSTATIS, #QEMIN, #QEMAX, and #QESUM.  Some of the details are filled in by the "Basic HAL/S Programming" course's section "HAL/S CSECTS".  That course explains that each HAL/S "compilation unit" is automatically assigned a 6-character "generic name" by removing all underscores and truncating to 6 characters.  (It's up to the programmer to insure that all of the generic names are unique!)  Since in the example we're using the block being compiled is named STATISTICS, the generic name is just "STATIS".  The compiler then generates names for the various control sections it likes to create by prefixing various 2-character strings to the generic name.  You can see the hopefully-full list of prefixes at the hyperlink just given, but the ones relevant to our particular example are:
  • #C — COMSUB code block, hence #CSTATIS 
  • #D — PROGRAM or COMSUB data, hence #DSTATIS
  • #Z — ZCON CSECT for COMSUB or REMOTE data, hence #ZSTATIS
  • #Q — ZCON for library routines, hence #QEMIN, #QEMAX, and #QESUM
Aside:  A COMSUB is a HAL/S compilation unit containing data or code (such our STATISTICS PROCEDURE) accessible by other compilation units.  ZCON code and REMOTE data are "far away" in the AP-101S address space from the code accessing them, and hence are accessed by different AP-101S instruction types from "nearby" data or code. The functions EMIN, EMAX, and ESUM are AP-101S assembly-language functions from the HAL/S runtime library.

Alas, the object file is also seen to have a number of items not directly observable in the original HAL/S code but rather presumably transparently generated by the HAL/S compiler itself; examples are the symbols STACK, STACKEND, HALS/FC, HALS/END, D24324, and so on.

Some of these items (symbols STACK, STACKEND) appear in the assembly-language source code for HAL/S runtime-library files like ACOS, and thus apparently relate to common features HAL/S uses internally.  Specifically, I believe the report may be telling us that the DSECT called STACK is 0x30 (STACKEND) halfwords in length. 

Aside: Recall that because of the nature of the HAL/S language, the HAL/S compiler can determine at compile time the maximum stack required by any compilation unit, although the assembler cannot do so for assembly-language  files.

Some other non-obvious symbols (D24324, T4041882) seem by my reckoning to relate to embedded datestamps and timestamps:
  • DYYDDD — the leading literal "D" presumably stands for "Date of compilation", while "YY" is the final 2 digits of the year, and "DDD" is the day of the year (1 through 365, or 366 for leap years).
  • TNNNNNNN — the leading literal "T" presumably stands for "Time of compilation", while "NNNNNNN" is the number of centiseconds since midnight (0 to 8640000).  It's TBD as to whether this number is left-padded to 7 digits.  (It's always 7 digits after 3 a.m., but I'm too lazy to get up between midnight and 3 a.m. just to check the padding.)
Still other symbols (such as HALS/FC and HALS/END) remain to be interpreted.  Of course, I can invent explanations for anything: Perhaps the presence of symbol HALS/FC is simply the compiler's way of telling us that the compilation was performed by the HAL/S-FC compiler (as opposed to the HAL/S-360 compiler).  But then, perhaps not.  Really, I haven't a clue!  If we're lucky, time will tell.  I suppose that if reverse engineering obsolete object-file formats were easy, those formats would be included as free bonus items in boxes of breakfast cereal.

Unexpected Hurdles

Here are some things I find that differ between AP-101 object files and System/360 object files, or at least which are unexpected from reading the System/360 documentation.  Some of these things you'll already have seen above (but without explanation) in the reports produced by reverse-engineering software developed partially using the inferences drawn below.  But reverse engineering is a messy, iterative process, so it's seldom easy to give a clean end-to-end narrative.

Unlike the expectation we (or at least I) would have, The #LSQRT section's addresses do not restart at 0, and instead the assembler continues to assign it addresses which continue incremented from the end of the SQRT section.  It is as if the assembler has already performed the step of partially linking these two sections of the code.  Why is this?

One might argue that while whoever coded the original assembler thought (for some unknown reason) that it was a good idea to present the assembly report in this fashion, it really doesn't make any difference:  As long as the object file produced by the assembler still represents these as independent sections, the linker can still reassign the addresses however it likes, with the result that after the final linking it may no longer be the case that the #LSQRT section immediately follows the SQRT section in memory.

Alas, that rationalization is incorrect.  Consider now the lines highlighted in green, and in particular this instruction:

LA	R1,A

For the assembler to assemble an instruction like this, in which a "base register" is not explicitly stated as an operand itself, it needs to somehow determine for itself the appropriate base register.  There are only two ways this can happen:

  1. If the operand (A in this case) is in that same code section as the instruction itself, in which case the appropriate base register is general register 3.
  2. Or else the operand is in an address block which has been made known the assembler via a previous USING pseudo-op.  The USING pseudo-op informs the assembler as to what assumption it can make about the contents of the various general registers.

In this example, though, neither of these conditions obtains.  As for condition #1, the instruction appears in the SQRT code section, but its operand (A) appears in a different section, namely #LSQRT.  As for condition #2, the only USING pseudo-op follows the instruction rather than precedes it, so there is no way for the assembler to make any assumptions about register contents.

How, then, has the assembler been able to assemble this LA instruction?  Well, perhaps there are possibilities that haven't occurred to me, but it appears to me that there's only one explanation:  The assembler really has already linked the SQRT and #LSQRT code sections, so that they are in a fixed relationship to each other, and in particular there is really no separate #LSQRT section as such in the object file output by the assembler.  I.e., #LSQRT has become just a label for an address within the SQRT section, and the actual linker program cannot change this at any later time.

Non-System/360 Datatype 0x84

To understand the comments below, it may help to refer back to the earlier example of a reverse-engineering report.

In so-called SYM records within an object file, each symbol representing "data" has an associated "datatype", consisting of a single-byte number.  For example, the datatype 0x00 represents character data (C), 0x04 represents hexadecimal data (X), 0x18 represents single-precision floating-point (E), and so on.  However, there's one datatype in AP-101S object files, 0x84, that doesn't appear in the available system/360 object-file documentation, leading us to speculate that it represents some AP-101S-only datatype not available on System/360. 

As it happens, there's discussion on our page about the modern AP-101 assembler that concerns the available datatypes.  That discussion points out a single AP-101S-specific datatype, Z, that's speculated to be unrelated to the System/360 datatype also designated as Z ("zoned decimal", 0x34).  For AP-101S, it was speculated that the Z datatype instead was something called a ZCON, which as kind of long-range address constant for items not present in nearby memory and hence addressable by only a restricted class of instructions. 

We are thus tempted to equate datatype 0x84 in SYM records with AP-101S ZCONs.  At the moment this is pure speculation that perhaps doesn't fit perfectly, but the reverse-engineering report nevertheless treats it as factual and reacts to these datatypes as follows:

  • 0x34 (System/360 zoned decimal) would be displayed in the reports as "z", but cannot actually appear.
  • 0x84 (possibly AP-101S ZCON) is displayed in the reports as "Z".

Encoding of Character Data in Object Decks

In System/360 object files, all character data is encoded in EBCDIC.  This is not true for AP-101 object files. Some text is encoded instead in the character set of the Space Shuttle's Display Electronics Unit (DEU).

The DEU character set is depicted in the table below.  It is an ASCII-like character set, in the sense that almost everywhere the printable characters or control codes overlap with printable ASCII characters or control codes, the numerical encoding matches ASCII as well.

PROGRAM Compilation Units

Object files generated from HAL/S source code containing a PROGRAM (i.e., the top-level compilation unit of a program, rather than a mere component of a program such as the PROCEDURE) have object files differing from the format described in the System/360 documentation and roughly covered in examples of reverse engineering above.

Such an object file begins with a block of records in the almost-System/360 format we've already described, but differs after an END record is encountered.  After an END record, we enter undocumented terra incognita, in which we can only rely upon whatever reverse engineering we can apply.

Consider this sample HAL/S source code for a PROGRAM based on one found in the book Programming in HAL/S:

  SIMPLE: PROGRAM;
          DECLARE PI CONSTANT (3.14159266);
          DECLARE R SCALAR;
          WRITE(6) 'Input R to compute PI R**2, or -1 to quit.';
          DO WHILE TRUE;
                  READ(5) R;
                  IF R < 0 THEN EXIT;
                  WRITE(6) 'R =', R, 'and PI R**2 =', PI R**2;
          END;
  CLOSE SIMPLE;

After compiling this file and parsing the object file, we find extraneous material following the END record.  If we pretend that this extraneous material consists of EBCDIC characters — which is only partially true —, it could be translated into ASCII essentially as follows:

" STACK $0SIMPLE                                                                 "
"cSYM      a      aaaSTART                                               I**20018"
"cESD      a   abSTART   aaaa aa $0SIMPLEcaaa                            I**20019"
"cTXT aaa  a   abU3aa                                                    I**20020"
"cRLD      a     acabaaac                                                I**20021"

We need to digress for a moment before trying to make some sense of this extraneous nonsense.  In the earlier discussion of the object-file format, I glossed over certain simple facts that I assumed you could glean from the System/360 documentation if you were actually interested.  But those facts become relevant to the discussion now, so let state them:

  • An object file is supposed to consist entirely of 80-byte records.
  • The first byte of each record is supposed to be the control character STX, i.e. 0x02 (or "c" in the representation above).
  • Records appear in the order of type SYM, type ESD, type TXT, type RLD, and type END.
  • The final 8 characters of each record have an undefined purpose, but traditionally are an "ident" string that increments with each line.  Note that in the example above I haven't shown the records preceding the extraneous material, but those omitted records have ident fields "I**2001" for the leading SYM record through "I**2017" at the END record.

So superficially at least, the extraneous material at the end of the object file starts with an aberrant record that does not begin with 0x02.  But after that aberrant record, the remaining extraneous material seems to follow the rules as if they comprised a completely separate object file.
