This page is a tour of the various kinds of files we've received that comprise the source code of Space Shuttle flight software like PASS and BFS, the  supporting analyses and reports concerning it, and the hierarchical folder structure in which that material resides.  This tour is not a theory of operation as to how that software functions, other than incidentally. 

Unfortunately, at present, I'm not only treating the original material as export-restricted by the U.S. International Traffic in Firearms Regulations (ITAR), but also am obliged to comply with private agreements about access restrictions that I had to make to obtain the material in the first place.  The privately-agreed restrictions are much more restrictive than ITAR:  Roughly speaking, if you are a U.S. citizen, residing in the U.S., are in the process of creating HAL/S or AP-101S development tools (compiler, assemblers, linkers, emulators, emulators for peripheral devices, and so on) or performing other preservation activity for which access to source material would facilitate the process, and are willing to comply with my demands for secrecy, then you may qualify for access.  Mere interest in the topic or speculation as to something you might like to do in the future if it tickles your fancy is not sufficient.  Sorry.  (I really am, you know!)

This means that I'm giving you a "tour" of material to which very, very few of you have any current current access.  Why am I wasting my own efforts on such a silly activity?  Well, most importantly, it remains my fervent wish that the access restrictions we're laboring under may be gradually reduced or even eliminated over time, in which case many more people will be able to see the material, at which point you'll want to see our hard-won conclusions about how this material is structured.  But in the meantime, having the explanations available may prove worthwhile for those lucky(?) few who do have access.

Introduction

At this writing, all of the material is contained in a single folder I call PFS/, which stands for Primary Flight Software, but that's a misnomer since all available PFS, FCOS, BFS, and BOS source materials are contained in this single folder.  This folder comprises a local-only git repository that for privacy reasons has no online version (at, say, GitHub).  Therefore, if you have access, you can make local changes to your copy of PFS/, but cannot directly push any changes back upstream to me.   Methods for approaching this indirectly can be discussed privately.

Material in PFS/ was received piecemeal, over the course of several years, before which it was apparently passed around in private hands, and is reportedly still in the hands of multiple unknown individuals.  I'm not privy to the provenance, or have knowledge of that prior access, so don't bother to ask me.

In so far as flight-software source code is concerned, it was all originally encoded in EBCDIC, but when received by me had at some time previously been re-encoded in 7-bit ASCII, in a generally-human-readable but not 100% perfect manner.  For example, EBCDIC characters not available in ASCII (¬ ¢) were sometimes encoded using other ASCII-based 8-bit "code pages" (but were garbage otherwise), and there were seemingly-random incursions of non-ASCII characters such as the numeric code 0x00.  To the best of my knowledge, all of those EBCDIC-to-ASCII errors have now been corrected in PFS/.

Additionally, one provision of the private agreement I made to obtain the material is that the source code would be anonymized by removal of all personally-identifying information.  The anonymization technique replaces all programmer names (or initials) with random-appearing strings like "^n", "\n", "^nn" (padded with blanks), or "\nn" (padded with blanks).  This desire for anonymization is related to requirements of the U.S. Digital Millennium Copyright Act (DMCIA), but appears to me to be a misguided application of the law ... after all, knowing that someone named (say) H. Potter did something to some file in June of 1985 is not revealing anything personal (such as an address, telephone number, etc.) about any individual person named H. Potter.  Files in some subdirectories of PFS/ have been anonymized, but not others (as explained below), and the databases for reversing the anonymization are contained in PFS/ as well.  Therefore, if the source material ever were to be released more generally, beyond the core need-to-know group — say, to any U.S. citizen qualifying under ITAR —, it might be impossible to do it by direct distribution of PFS/.  Rather:

  • If the current requirement for anonymization is still in force, then the distributed version would have to remove the anonymization databases, and the git history would have to be scrubbed of all non-anonymized source-code files.
  • If anonymization could be relaxed, then the source-code files could be de-anonymized (my preference!) before distribution, although PFS/ could be distributed as-is in that situation.  The file PFS/README.md explains the anonymization process.  The de-anonymization process has not been developed, but should be reliable if implemented properly.

Layout of Flight-Software Folders

Folders containing flight-software source code are laid out with the following subdirectories:

  • SSSRC/ contains "system software" such as FCOS and possibly additional software that's always present regardless of the memory configuration.  Typically AP-101S assembly language and HAL/S source-code files.
  • APPLSRC/ contains "application software" that runs atop the system software.  Such software may be present for some flight phases and not others, and could potentially reside at different memory addresses in different memory configurations.  Typically HAL/S source-code files and input files for the Display Format Generator (DFG) preprocessor, which converts these files to HAL/S.
  • INCL80/ contains fragmentary HAL/S source-code files — i.e., files which which don't by themselves comprise a complete "compilation unit" and therefore cannot themselves to compiled as-is by a HAL/S compiler —, but which can be included in other HAL/S source-code files via the compiler directive "D INCLUDE filename".  Such files typically contain STRUCTURE templates and REPLACE (macro-definition) statements.
  • MLIB80/ contains assembly-language files of two types:  those includable in other assembly-language files explicitly via the assembly-language COPY pseudo-op, and those containing macro definitions expected always to be available to the assembler.  A file (MACROFILES.txt) is present in the folder to specify which of these two categories each source-code file falls into.
  • CON80/ contains files of "linkage-editor command cards", which are basically sets of complex instructions to the linker, telling it how to handle each object-code files created by the HAL/S compiler or AP-101S assembler.  This is discussed in more detail later.
  • PDTIN/ is TBD.  May stand for Parameter Data Tape INstructions, and seem to be associated with specific COMPOOLs.  Perhaps this relates to how telemetry downlists were stored on tape?
  • PSFIN/ is TBD.  I merely note that these files are very similar in nature to the files in PDTIN/, and hence may relate to some similar use.  Perhaps this relates to how downlisted data was formatted for viewing?
  • TOOLIN/ seems to relate (at least partially) to the "language subset" feature of the HAL/S language.  Apparently, it works something like this:  The so-called Software Architecture Review Board (SARB) established a set of software-programming standards to which the flight software was required to comply unless SARB granted exceptions for them.  Compliance was enforced via code reviews.  The language-subset feature facilitated the compliance reviews by causing the HAL/S compiler to reject disallowed features.  In so far as language subsets are concerned, TOOLIN/ folder essentially is a library of constraints on the HAL/S compiler.  I don't actually understand much more than I've just said — if even that much —, and I'd recommend reading TOOLIN/$$SUBSET and/or the software-programming standards document for more information.  As of this writing, no attempt has been made to exercise the language-subset feature, so it may not even be implemented correctly in the HAL/S compilers as ported for modern use.  But for all I know, TOOLIN/ may have additional files unrelated to language subsets.

File Types

  • *.hal:  HAL/S source code
  • *.asm:  AP-101S assembly-language source code
  • *.dfg:  Display Format Generator (DFG) source code.  The display format generator is a preprocessor that converts *.dfg to *.hal
  • *.con:  Linkage-editor control-card files.  They provide instructions to the AP-101S linker.
  • *.fcm:  An image of AP-101S memory, typically comprising a complete, executable array of software.
  • *.ASC, *.SRC, or no filename extension:  Usually, source material exactly as received, often as reports that need to be processed in order to extract actual source code or executable code.

Memory Configurations

BFS.SRC/, "BFS.SRC as received"/, and BFS.rpts/

The contents of these directories are believed to provide perhaps 10% of the source code for the Backup Flight Software (BFS) and Backup Operating System (BOS) associated with release OI-34.1, for the reasons described here., and to be compatible with compiler HAL/S-BFC 16.1.  Since the version of the HAL/S compiler available to use is 17.0, there should be compatibility between the code and the compiler.

It is unclear, though, how much of BFS source code is available versus how much of BOS source code is available, given that one of the source-code files (in BFS.SRC/SSSRC/) is actually called /BOS.asm, and that all of BOS.asm's direct dependencies (ENTRYS.asm, EQU.asm, and so on) are actually available (in BFS.SRC/MLIB80/).  ENTRYS.asm does list many EXTRN symbols, but at least upon superficial examination these appear to be variables rather than subroutines. 

In summary, it remains possible at this writing that the entirety of BOS source code, other than allocations of EXTRN'd variables, may indeed be present.  This is something that should be investigated.
